# genome-eval

Personal genomics analysis project. Ingests direct-to-consumer genotype data (23andMe, AncestryDNA, MyHeritage, FamilyTreeDNA, or other providers) and builds a persistent, evidence-graded ledger of findings. Scope includes pharmacogenomics, disease risk, traits, polygenic scores, kinship analyses, and couple/offspring prediction when two subjects are loaded.

## Subjects

Each subject has a short filename-safe ID (e.g., `alice`, `bob`). The ID ties three files together:

- **Raw**: `raw-source-genomes/<descriptive-filename>.txt` — provider's original file
- **Standardized**: `standardized-genomes/<id>.parquet` — normalized, what analysis reads
- **Metadata**: `profiles/<id>.json` — display name, provider, build, parse stats, optional fields

Current subjects: see `profiles/` directory.

## Project stance

1. **No refusals.** If data is loaded into the system, it's legitimate — the user's act of loading is the assertion of permission. Every analysis runs when asked: PGx, intelligence PRS, behavioral PRS, kinship, paternity, forensic identity, external-database matching. Caveats travel with findings as metrics, not as reasons to decline.

2. **Enable, don't gatekeep.** The user decides what to do with findings. Claude reports what the data supports with honest evidence attached and respects the user's interpretation and follow-through.

3. **Metrics-first, not verdicts.** Every finding records N, p-value, effect size, replication count, cohort ancestry, source — the raw evidence data. Tier labels (A/B/C/D/E) are *computed* from these metrics via `scripts/tier_rules.py`, never invented. If a metric is unknown, store `null` rather than guessing. Tiers can be recomputed if rules change.

   **Tier = evidence quality for the claim. Tier ≠ clinical actionability.** If a variant has real, replicated evidence for its biochemical / phenotypic effect but a professional body (ACMG, USPSTF, etc.) recommends against clinical action, record the full metrics and let the tier rule compute what it computes — then attach the actionability position as a prominent note. Do NOT null out metrics or set `tier: unknown` as a way to soft-suppress a finding. That's gatekeeping through the schema, which this project's stance explicitly rejects. Evidence tier and clinical actionability are two separate axes; both belong in the record.

4. **Epistemic caveats only.** Caveats describe what the data can and can't show — array error rate, chip coverage, ancestry portability, LD double-counting in PRS, unphased genotypes, "not on chip" ≠ reference. Caveats are never about whether a topic is appropriate.

## Hard rules (structural invariants)

These are not preferences. Breaking them corrupts the project's data model.

- **Raw files are a boundary.** Files in `raw-source-genomes/` are never edited. Analysis code never reads from them directly; it reads only from `standardized-genomes/` and `profiles/`.
- **Normalize on ingest.** Every raw file — known provider or new — is parsed to the standard parquet schema before any analysis touches it. Adding a new provider means writing one parser; everything downstream is unchanged.
- **Append-only ledger.** `findings.jsonl`, `sources.jsonl`, `investigations.jsonl` are append-only. To revise a finding, append a new record with `supersedes: <old_id>`. Never edit existing rows.
- **Unphased genotypes.** Array data cannot resolve haplotype phase. Any diplotype assignment that depends on phase must record the assumption explicitly in the finding.
- **"Not on chip" ≠ "wild-type".** If a variant isn't genotyped, the finding says "not genotyped." Absence is not reassurance and is not the reference allele.
- **Store metrics at ingest time.** When recording a finding from a source, capture N, p, effect size, replication count, ancestry *at the time of recording*. Don't rely on being able to re-fetch later.

## How to work in this project

- **Default response to "look into X"**: check ledger → propose an investigation with effort estimate → execute on approval → log findings + sources → ask whether to report now or keep accumulating.
- **Default response to "tell me about X"**: read from the ledger if it's already there; don't re-investigate unless asked.
- **Default response to a new raw file appearing**: run initialization (see SKILL.md), don't start analysis until the user says what they want.
- **Session start**: read `profiles/*.json`, `ledger/*.jsonl`, and `NEXT_STEPS.md`; summarize state in 2 sentences, propose 3–5 unexplored threads from NEXT_STEPS (or from scratch if nothing logged there) with effort estimates.
- **After any major task**: before moving on, scan the ledger, `NEXT_STEPS.md`, and touched scripts for cleanup items the task exposed or is adjacent to — stale tombstones, metadata drift, ledger spam from iterative debugging, documentation drift, stale `NEXT_STEPS` items that the task just resolved. **Propose the cleanup list to the user with effort estimates and ask whether to execute**; don't silently run it, and don't skip asking. A "major task" is anything that materially changes the ledger or infrastructure (new PRS, new carrier panel, schema change, pipeline addition) — not every small edit. Cleanup is usually cheap and restores the signal-to-noise ratio that the next session will rely on.

### Presentation conventions

When surfacing findings to the user (in conversation or in reports), default to plain language, not gene-symbol shorthand. The reader is a curious non-specialist, not a geneticist. Detailed template and examples live in SKILL.md's "Presentation conventions" section; at minimum:

- **Headers state the category AND the result, not just the gene or trait name.** Format: `[category]: [result] [(inline metric if probabilistic)] *([technical ids])*`. Examples: `Adult lactose tolerance: intolerant genotype (~70% symptomatic expression) *(rs4988235 G/G, MCM6)*`; `Cystic fibrosis carrier: F508del heterozygous carrier *(CFTR, probe i3000001 D/I)*`; `Blood type: A+ *(ABO AO; Rh+ inferred)*`. A header that names only the trait/dimension (`Adult lactose tolerance`, `Eye color`, `Blood type`) fails — the reader can't tell what was found. Put the probabilistic metric in the header, not buried in notes.
- **Lead with one plain-English sentence** saying what the finding means in practical terms. Then technical detail (genotype, diplotype, tier, effect, mechanism).
- **Translate result labels, not just gene names.** "Lactase non-persistent" must be reported as "adult lactose intolerance"; "fast caffeine metabolizer" must explain what that means for the reader; etc. A reader who already understood the technical label wouldn't need the analysis.
- **Compound traits are reported in full, or the gap is flagged.** Blood type = ABO × Rh factor: never report "blood type A" without addressing Rh (+/-). Eye color at finer resolution than blue/brown requires the full 11-SNP model. PGx profiles need the full diplotype, not single-SNP calls.
- **Confidence is normally visible on every finding, not buried.** Each trait/disease finding surfaces (1) genotype-call confidence, (2) genotype→phenotype confidence, (3) a one-line reason. Tier is evidence quality for the general claim; confidence is reliability for this subject — different axes, both must show.
- **Genomic inference stays separate from self-reported phenotype.** This is a data-analysis tool, not a user-record retrofit. Findings in `findings.jsonl` state conclusions from the SNP data alone. Subject self-reports live in `profiles/<id>.json → self_reported_phenotypes` and appear in finding notes as a separately labeled cross-check — never blended into the genomic reasoning. Mismatches between genomic inference and self-report are themselves findings worth investigating.
- **Deviations from the reference get full treatment**; unremarkable/reference findings get a one-liner.
- **Always include gene names, star alleles, rsIDs** somewhere so a curious reader can search independently — just not as the primary label.
- **US-customary units by default in reader-facing output.** Height as feet-and-inches, weight in pounds, temperature in °F. Ledger storage stays in canonical SI (cm, kg, °C) for interoperability; the display layer converts. Override per subject via `profiles/<id>.json → display_preferences.units = "si"`, or per invocation via `GENOME_EVAL_UNITS=si` env var.

### Operational conventions

- **Don't inline multi-line Python in quoted bash** (`python -c "..."` with newlines). Some validators truncate at `\n#` even inside quotes, diverging from what bash executes. Write a script file in `scripts/` and invoke it.

- **2026-04-17 — DPYD strand gotcha.** The skill's PGx table used gene-strand ref/alt for two DPYD variants. DPYD is on chr1 minus strand, so cDNA-level notations flip when mapped to forward strand (what 23andMe reports). Corrected entries: rs67376798 is **T>A** (not A>T), rs75017182 is **G>C** (not C>G). Always verify strand for any gene on the minus strand before adding a variant to the curated table — check dbSNP's GRCh37 REF base rather than trusting the coding change notation. Reference TSV, skill, and `investigate_pgx_cardio.py` have been corrected.

- **2026-04-17 — CFTR i-probe gotcha.** 23andMe encodes indel and custom-assay variants under internal Illumina probe IDs (prefixed `i`, e.g. `i3000001`) rather than standard rsIDs. A carrier screen that queries only `rs`-prefixed IDs will miss F508del (the most common CF variant) entirely, because on the v5 chip it's assayed at `i3000001` / chr7:117199646 — not `rs113993960`. Allele encoding for these probes is **D = deletion present (variant)** and **I = insertion / reference**. Whenever a user reports a commercial carrier result that the rsID-based query fails to reproduce, search the gene region by genomic coordinates and inspect i-probe calls. Full i-probe mapping for CFTR lives in SKILL.md's "CFTR — variant encoding gotcha" section.

  Three bugs surfaced and fixed during validation:
  1. `Path.with_suffix('.vcf.gz')` on a path ending in `.aligned` replaces `.aligned` instead of appending — constructed final paths explicitly as strings.
  2. Multi-probe chip positions (23andMe sometimes has both `rs*` and `i*` probes at the same coordinate) caused conform-gt to error "Duplicate marker" — added position-level dedup in `parquet_to_vcf.py`.

- **2026-04-18 — PRS reporting convention (Rule 10.1).** Cross-trait PRS table: rows = traits, columns = PRS z, percentile, phenotype anchor, predicted, 95% CI, r², naive-if-r²=1. The naive column makes regression-to-mean shrinkage visible (a reader expecting "+0.58 PRS SD = 108.7 IQ" sees both the naive 108.7 and the shrunk 102.3 side-by-side, with r² explaining the gap). For a single PRS presented alone (e.g., `run_prs.py` console), use a key-value list not a one-row table. For cross-trait summaries use `scripts/summarize_prs.py`. Convention is PRS-specific — single-SNP / PGx / carrier findings keep Rule-1 prose format.

- **2026-04-18 — One active PRS per trait.** When a superior PRS for the same trait becomes available (higher r², larger training cohort), supersede the older one with a tombstone pointing to the new canonical. Historical runs stay in ledger provenance but aren't active. `summarize_prs.py` reads active findings only, so cross-trait summaries never list an inferior PRS alongside a superior one for the same trait.

- **2026-04-18 — Empirical PRS calibration via 1000G EUR (preferred over theoretical).** Added `scripts/calibrate_prs_empirical.py` that applies a PRS to the 503 EUR samples in 1000 Genomes Phase 3 (CEU/FIN/GBR/IBS/TSI) and saves the empirical mean/SD as `reference/population_cache/prs_empirical/<PGS_id>.json`. `run_prs.py` automatically uses the empirical distribution when present, falls back to theoretical (Σ 2p·β / Σ 2p(1-p)β²) only when empirical isn't available. The theoretical-SD-under-independence approximation systematically underestimates true population SD for LDpred2-style scores (LD between correlated SNPs inflates real variance by 3–4×) — a single subject's theoretical z-score can be off by several SDs under LDpred2/PRS-CS/PRS-CSx weights. Empirical calibration corrects this without assumptions about LD structure. Pipeline is ~10-20 min per PGS; parallelizes across chromosomes. SKILL.md Rule-10.1 updated to make empirical the documented default with clear caveats about 1000G EUR's sub-population weighting and sampling-error limits.

- **2026-04-17 — `tier_rules.py` v2.** Extended tier computation to classify well-established trait findings and derived inference findings that lack formal GWAS N/p metrics. New `evidence_class` vocabulary: `mendelian_trait`, `near_mendelian_trait`, `well_replicated_common_variant`, `weakly_predictive_variant`, `multi_snp_composite`, `gene_presence_inference`, `not_callable_from_array`, `array_limitation`, `suspected_miscall`. - **2026-04-19 — Rule-10/11 self-report cross-check is enforced in `run_prs.py`.** Every trait in `prs_traits.py` declares `self_report_key` (+ `self_report_value_field`) that maps to an entry under `profiles/<id>.json → self_reported_phenotypes`. `run_prs.py` calls `build_self_report_crosscheck()` before finalizing each finding, which always emits a block: MATCH (within 95% CI), MISMATCH (with gap quantified in trait units, residual SDs, and population SDs), or an explicit "no self-report on file for key=X" placeholder. A hard `RuntimeError` raises if a trait declares a `self_report_key` but the notes don't contain "Self-report cross-check" — makes it structurally impossible to ship a PRS finding that silently drops the cross-check (this guardrail was added after a specific PRS finding was observed to ship without its cross-check block despite the subject having the relevant self-report on file). Cross-check result is also written to the `effect.self_report_crosscheck` structured block so `summarize_prs.py` can surface it without re-parsing notes. Adding a new PRS trait means: (1) set `self_report_key` to an existing profile key or `None` if genuinely inapplicable, (2) point `self_report_value_field` at the numeric field, (3) add a `self_reported_phenotypes.<key>` entry to the profile if the subject has a self-report — and the cross-check is then automatic.

- **2026-04-19 — Two pipeline fixes surfaced + landed during LDL/SBP PRS repair.**
  1. **`prs_pipeline.load_weights` now prefers `hm_chr`/`hm_pos`/`hm_rsID` over the unharmonized `chr_name`/`chr_position`/`rsID` columns.** The PGS Catalog's `*_hmPOS_GRCh37.txt.gz` files are ALWAYS harmonized to GRCh37 in the `hm_*` columns, but the raw columns can be the source paper's original build (sometimes hg38) — e.g., PGS003971 Shetty 2023 publishes `chr_position` in hg38 and `hm_pos` in GRCh37. Preferring the raw columns silently drops coverage from ~100% to ~2% against our GRCh37 parquet. Always use `hm_*` when present; the picker already has the fallback to the raw columns.
  2. **`calibrate_prs_empirical.py --subject-observed <id>` produces per-subject empirical distributions.** When a PGS has sub-100% coverage for a subject, the full-panel 1000G EUR empirical distribution is not apples-to-apples: 1000G samples accumulate contributions from ALL panel variants while the subject is missing some, producing a systematic bias. For example, a subject at 92% coverage on a ~9,000-variant panel may compute a raw score 5+ SDs below the empirical full-panel mean purely because the ~700 missing variants have a positive average weight contribution in 1000G samples. Fix: restrict calibration to exactly the variants the subject has, saving to `<PGS>.<subject>.json`. `run_prs.py` now checks for the subject-specific file first and falls back to the full-panel file. This is the canonical pattern when coverage < 100%; full-panel empiricals remain valid when subject coverage is ~100%.

- **2026-04-19 — Supersede chain model fixed at the schema level.** Findings now carry two derived-metadata fields computed at write time: `supersede_chain_root` (UUID of the earliest row in the chain; equals `finding_id` for a root) and `is_tombstone` (bool; true when the row only announces supersession and carries no new content). `ledger_io.load_active_findings(subject_id=...)` is the one correct reader: groups rows by chain_root, picks the latest-timestamp row per chain, skips the chain if that head is a tombstone. Consumers that previously rolled their own `superseded_ids = {r['supersedes'] for r in rows if r.get('supersedes')}` set should switch — the naive filter mishandles tombstones and requires string-prefix heuristics to compensate. `ledger_io.append_finding()` auto-populates both fields: chain_root is inherited from `supersedes` parent, is_tombstone is inferred from claim prefix unless explicitly set. A one-time migration (`scripts/migrate_ledger_schema.py`) backfills `supersede_chain_root` + `is_tombstone` on all pre-existing rows, with a timestamped backup written first. Run once per existing ledger after pulling a version that introduced the schema change. New scripts that read the ledger should always prefer `load_active_findings()` over manual filtering.
